# Базовые окружения маршрутизатора и сети

## Базовое окружение маршрутизатора
Параметризованное окружение (`DATA_WIDTH`, `PORTS_NUM`) для автономной верификации маршрутизатора

Окружение имитирует подключенный в общую сеть маршрутизатор, подавая данные на каждый порт

### Основная идея

Любой маршрутизатор (под любую топологию и интерфейс) можно описать зная только количество его портов и размерность передаваемых данных  
В нотации SystemVerilog каждый порт описывается как:  

    logic [DATA_WIDTH-1:0] data;
    bit valid;
    bit ready;
    int destination;

- Поле `destination` может описывать маршрутизацию в любой топологии; для реальных маршрутизаторов это поле необходимо будет маппить на реальные сигналы адреса.
- Поля `valid`, `ready` - базовый handshake, сообщающий о валидности данных на порте и о готовности конкретного порта принять данные.  

Общий структура транзакции:  

    base_router_port [PORTS_NUM-1:0] ports;
    bit is_input;

Это массив всех портов и флаг, указывающий, являются ли данные входными или выходными.

### Архитектура
Окружение написано в соотвествие с методологией UVM и содержит в себе:
1. `base_router_item` - транзакция
2. `base_router_config` - конфигурация
3. `base_router_sequencer` - сиквенсер
4. `base_router_driver` - драйвер (переопределяется)
5. `base_router_monitor` - монитор (переопределяется) 
4. `base_router_agent` - агент
5. `base_router_scoreboard` - скорборд
6. `base_router_coverage` - сборщик покрытия
7. `base_router_env` - окружение верхнего уровня

### Проверка функциональности:
Отслеживается статистика handshake на каждом порту, подсчитывая количество успешно принятых(`accepted`), потерянных(`dropped`), отправленных(`ejected`) и застрявших(`stalled`) пакетов. 


### Функциональное покрытие:
1. Адрес назначение (от `0` до `ROUTERS_NUM-1`)
2. Порт источника
3. Тип пакета: входной или выходной

Функциональное покрытие = декартово произведение всех точек покрытия


### Тесты
Реализован один настраиваемый через конфигурацию тест:
`base_router_rand_test`  
Параметры:
1. `PACKETS_COUNT` - количество отправляемых пакетов
2. `ROUTERS_NUM` - количество маршрутизаторов в сети
3. `SRC_PORT` - порт с которого должны посылаться данные (от `0` до `PORTS_NUM-1`, `-1` если порт-источник выбирается случайно)
4. `DEST_ROUTER` - адрес посылки (от `0` до `ROUTERS_NUM-1`, `-1` если целевой маршрутизатор выбирается случайно)
5. `STRESS_MODE` - режим(`0`/`1`), в котором со всех портов идут валидные данные

На каждой итерации теста, окружение получает транзакцию для каждого порта (либо idle-транзакцию при отсутствие валидных данных)

### Адаптации под конкретный маршрутизатор
Задача верификатора, использующего данное базовое окружение для реального дизайна: 
1. В драйвере - смаппить сигналы `base_router_item` на сигналы конкретного интерфейс
2. В мониторе - смаппить сигналы интерфейса обратно на `base_router_item`
3. Создать наследника нужного теста, указав конкретные параметры через #(`DATA_WIDTH`, `PORTS_NUM`)

Переопределение драйвера и монитора выполняется через механизм UVM-фабрики в наследнике.  
При необходимости легко добавляются специфичные для конкретного маршрутизатора тесты, блоки проверок, сборщики покрытия и другие верификационные компоненты.

## Базовое окружение сети
Параметризованное окружение (`DATA_WIDTH`, `ROUTER_NUM`) для автономной/системной верификации сети на кристалле

Окружение имитирует/мониторит СФ-блоки, подключенные к каждому локальному порту

### Основная идея

Сеть представляет собой набор маршрутизаторов, каждый из которых имеет локальный порт, подключенный к какому-либо блоку.   
Любую сеть (любой топологии и интерфейса) можно описать как массив локальных портов всех маршрутизаторов   
В нотации SystemVerilog это можно описать как:  

    base_router_port local_ports[ROUTERS_NUM];
    bit is_input;

### Архитектура
Окружение написано в соотвествие с методологией UVM и содержит в себе:
1. `base_noc_item` - транзакция
2. `base_noc_config` - конфигурация
3. `base_noc_sequencer` - сиквенсер
4. `base_noc_driver` - драйвер (переопределяется)
5. `base_noc_monitor` - монитор (переопределяется) 
4. `base_noc_agent` - агент
5. `base_noc_scoreboard` - скорборд
6. `base_noc_coverage` - сборщик покрытия
7. `base_noc_env` - окружение верхнего уровня

### Проверка функциональности:
Отслеживается статистика handshake на каждом локальном порту, подсчитывая количество успешно принятых(`accepted`), потерянных(`dropped`), отправленных(`ejected`) и застрявших(`stalled`) пакетов. 


### Функциональное покрытие:
1. Адрес назначение (от `0` до `ROUTERS_NUM-1`)
2. Адрес источника (от `0` до `ROUTERS_NUM-1`)
3. Тип пакета: входной или выходной

Функциональное покрытие = декартово произведение всех точек покрытия


### Тесты
Реализован один настраиваемый через конфигурацию тест:
`base_noc_rand_test`  
Параметры:
1. `PACKETS_COUNT` - количество отправляемых пакетов
2. `SRC_ROUTER` - маршрутизатор с которого должны посылаться данные (от `0` до `ROUTERS_NUM-1`, `-1` если источник выбирается случайно)
3. `DEST_ROUTER` - адрес посылки (от `0` до `ROUTERS_NUM-1`, `-1` если целевой маршрутизатор выбирается случайно)
4. `STRESS_MODE` - режим(`0`/`1`), в котором со всех маршрутизаторов идут валидные данные

На каждой итерации теста, окружение получает транзакцию для каждого локального порта (либо idle-транзакцию при отсутствие валидных данных)

Если окружение системное (DUT уже содержит подключенные к локальным портам СФ-блоки), то достаточно при запуске теста указать AGENT_ACTIVE=0, тогда окружение переходит в режим мониторинга и проверки, никакие сигналы на интерфейс выставляться не будут
